# Codex Commenting Prompts Guide

## Purpose

This guide captures the decisions made in this conversation about improving code comments generated by Codex **module by module**, without changing the global prompt library first.

The goal is to keep working incrementally:
- one module at a time
- focused prompt per task
- comments requested explicitly
- comments always in English
- no unrelated refactors

This approach was preferred over doing a broad update first in `firmware/04_codex_prompts.md`.

---

## Agreed working approach

Instead of relying on a global documentation change, the preferred workflow is:

1. pick one module
2. ask Codex to update comments only in that module
3. keep behavior unchanged
4. keep naming unchanged unless the task explicitly asks otherwise
5. require useful comments in English
6. avoid redundant comments that only restate obvious code

This keeps the patch small and ownership clear.

---

## Comment policy to ask from Codex

When asking Codex to improve comments in a module, the prompt should request:

- comments in English only
- useful comments, not comment noise
- comments that explain:
  - intent
  - ownership boundary
  - hardware constraints
  - timing assumptions
  - non-obvious implementation choices
- preservation of already useful comments
- improvement of weak, unclear, or missing comments only where local to the requested module

The prompt should also explicitly avoid:

- line-by-line narration of obvious code
- unrelated cleanup
- behavior changes
- renaming unrelated symbols
- broad formatting churn

## Reusable prompt template

Use this when you want Codex to improve comments in a specific module.

```text
Work in `firmware/`.

Task:
Improve code comments in <target module>.

Goal:
Add or refine useful code comments in English for this module, without changing intended behavior.

References:
- firmware/AGENT_ENTRY.md
- firmware/STYLE.md
- firmware/style-examples/README.md
- firmware/04_codex_prompts.md

Constraints:
- comments/documentation-focused change only
- no behavior expansion
- no unrelated cleanup
- do not rename unrelated symbols
- keep file ownership clear
- preserve existing useful comments
- improve weak or unclear comments only where local to this module
- comments must be in English
- prefer meaningful comments over verbose narration
- avoid comments that simply restate obvious code line by line

Expected result:
- minimal patch
- improved readability
- unchanged intended behavior
- clearer English comments where they add value
```

## Stronger variant

Use this version when Codex tends to add too few comments.

```text
Work in `firmware/`.

Task:
Improve code comments in <target module>.

Goal:
Add more useful code comments in English than the module currently has, but only where they improve maintainability and understanding.

References:
- firmware/AGENT_ENTRY.md
- firmware/STYLE.md
- firmware/style-examples/README.md
- firmware/04_codex_prompts.md

Constraints:
- comments/documentation-focused change only
- no behavior expansion
- no unrelated cleanup
- do not rename unrelated symbols
- keep file ownership clear
- preserve existing useful comments
- add comments where intent, ownership, constraints, or hardware interaction are not obvious
- comments must be in English
- avoid line-by-line narration of trivial code
- avoid noisy or decorative comments

Expected result:
- minimal patch
- improved maintainability
- unchanged intended behavior
- noticeably better English comments in the requested module
```

## When to use this workflow

This module-by-module comment workflow is a good fit when:

- you want tight review control
- you want small diffs
- you do not want a repo-wide documentation churn
- you want to improve comments gradually while continuing normal refactors
- Codex responds better to narrowly scoped tasks than to broad style-policy edits

---

## Practical recommendation

Use the basic template by default.

If Codex still writes too few comments, switch to the stronger variant and mention exactly what kind of comment is missing, for example:
- ownership comments
- IRQ/path comments
- timing comments
- hardware constraint comments
- why-this-way comments

That tends to work better than asking for “more comments” in the abstract.
